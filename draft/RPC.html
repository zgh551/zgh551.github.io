RPC 分布式部署方案

概述

RPC是远程过程调用(Remote Procedure Call)的缩写形式，简而言之 ，就是 “像调用本地方法一样调用远程方法”。RPC可基于 HTTP、TCP或UDP 协议，Web Service就是基于 HTTP 协议的 RPC，它具有良好的跨平台性，但其性能却不如基于TCP 协议的 RPC。影响 RPC 性能的因素包括两方面，一个是传输方式，另一个是序列化。

  1. 数据传输协议 TCP 是传输层协议，HTTP 是应用层协议，而传输层较应用层更加底层，在数据传输方面，越底层越快，因此，在一般情况下，TCP 一定比 HTTP 快。
  2. 序列化方法 目前拥有一系列优秀的序列化框架，比如：Protobuf、Kryo、Hessian、Jackson 等，常见序列化格式为Binary、Compact、JSON、Multiplexed 等。

RPC开源框架介绍

gRPC

gRPC是一个高性能、通用的开源RPC框架，其由Google 2015年主要面向移动应用开发并基于HTTP/2协议标准而设计，基于ProtoBuf序列化协议开发，且支持众多开发语言。其基本原理是通过 IDL(Interface Definition Language)文件定义服务接口的参数和返回值类型，然后通过代码生成程序生成服务端和客户端的具体实现代码，这样在 gRPC 里，客户端应用可以像调用本地对象一样调用另一台服务器上对应的方法。 它的主要特性包括三个方面。

  • 通信协议：采用了 HTTP/2，因为 HTTP/2 提供了连接复用、双向流、服务器推送、请求优先级、首部压缩等机制。
  • Interface Definition Language(IDL)：IDL使用了ProtoBuf，ProtoBuf 是由 Google 开发的一种数据序列化协议，它的压缩和传输效率极高，语法也简单。
  • 多语言支持：能够基于多种语言自动生成对应语言的客户端和服务端的代码。

Thrift

Thrift是一种轻量级的跨语言的端对端RPC通信方案，支持多达 25 种编程语言。为了支持多种语言，跟 gRPC 一样，Thrift 也有一套自己的接口定义语言 IDL，可以通过代码生成系统，生成各种编程语言的 Client 端和 Server 端的 SDK 代码，这样就保证了不同语言之间可以相互通信。 下图可以看出 Thrift RPC 框架的主要特性如下：

  • 支持多种序列化格式：如 Binary、Compact、JSON、Multiplexed 等；
  • 支持多种通信方式：如 Socket、Framed、File、Memory、zlib 等；
  • 服务端支持多种处理方式：如 Simple 、Thread Pool、Non-Blocking 等；

Dubbo

Dubbo是阿里集团开源的一个极为出名的 RPC 框架，在很多互联网公司和企业应用中广泛使用。协议和序列化框架都可以插拔是极其鲜明的特色。目前只支持 Java 语言。

RPC原理

RPC的一般性调用流程如下：

  1. 客户端(client)：服务调用方，采用本地调用方式调用远程服务；
  2. 客户端存根(client stub)：存放服务端的地址和端口信息，负责将客户端请求的方法、参数等按照特定协议进行组装，然后对组装的包进行序列化操作，转化成能够进行网络传输的二进制数据流，最后通过网络传输发送给服务端。
  3. 服务端存根(server stub)：接收客户端发送过来的请求消息并解包，对二进制数据流进行反序列化操作，最后再调用本地服务进行处理。
  4. 服务端(server)：服务的真正执行者；
  5. Network Service: 网络底层传输协议，可以采用TCP、UDP或HTTP。

RPC调用方式

主流的RPC框架一般会提供四种调用方式，分别为同步(sync)、异步(async)、回调(callback)和单向(oneway)。

同步调用(Sync)

客户端线程发起RPC调用后，当前线程会一直阻塞，直至服务端返回结果或者处理超时异常。

异步调用(Async)

客户端发起调用后不会再阻塞等待，而是拿到 RPC 框架返回的 Future 对象，该对象可以在将来获取异步执行的结果。调用结果会被服务端缓存，客户端自行决定后续何时获取返回结果。当客户端主动获取结果时，该过程是阻塞等待的。

回调调用(Callback)

客户端发起调用时，将 Callback 对象传递给 RPC 框架，无须同步等待返回结果，直接返回。当获取到服务端响应结果或者超时异常后，再执行用户注册的 Callback 回调。

单向调用(Oneway)

客户端发起请求之后直接返回，忽略返回结果。

TVM的RPC介绍

如上图所示，TVM的RPC组件被设计为如下四个主要模块：

RPCSession模块

RPCSession是一个纯抽象接口，包含一组需要实现的功能函数，具体的实现可以由不同的提供者完成。

  1. LocalSession：本地机器是一种特殊情况 ，它将所有调用委托给本地runtime。
  2. ClientSession：RPC客户端是另一个子类，它将会话调用委托给另一端RPCEndPoint提供的会话。

RPCEndpoint模块

使用通信通道将当前端点的 RPCSession 请求转发到另一个远程的RPCSession端点。

RPCChannel模块

RPCChannel同样是一个纯抽象接口，用于建立两个端点间的通信通道，目前TVM已基于socket实现。对于不支持联网的情况，可以添加更多的通道，比如 stdio。

  1. SockChannel：基于socket协议建立通信通道；
  2. PipeChannel：基于Linux管道的通信通道；
  3. CallbackChannel：带回调函数的通信通道；
  4. RPCChannelLogging：记录发送命令的通道实现；

RPCModule模块

在 TVM 的Runtime API 中将 RPCSession 暴露为 rpc 设备。

目前TVM的RPC功能可以支持graph和vm执行器。

TVM的RPC工作模式

直连模式

首先需要在服务端搭建PRC服务，然后客户端通过Connect函数直接与服务端建立RPC连接。

Tracker模式

RPC Tracker模式包含三个重要组件，分别是客户端、服务端和Tracker服务，三个组件的交互逻辑如下：

  • 服务端：将Tracker服务作为注册中心，注册服务端的地址、端口和key等信息。
  • 客户端：首先向Tracker服务请求特定的key值，然后Tracker服务将返回与key值绑定的服务端的IP地址，最后客户端基于返回的IP地址与服务端建立RPC连接。
  • Tracker服务：监听9010端口上所有IP的注册请求，将key与ip地址绑定；响应请求的key值，并返回与之绑定的IP地址。

分布式部署情形

如下图所示，是芯片内部的网络通信链路，不同域之间的虚拟网卡通过交换机连接，不同子系统之间通过PCIE连接。分布式部署的各类情形主要存在于不同域、不同子系统和不同SOC之间。

同一子系统不同域之间的调度

假设NPU设备映射在QNX域，当Android域的应用需要调用NPU设备进行模型推理时，可以通过RPC方式远程调用QNX域的NPU设备进行推理。

不同子系统不同域之间的调度

假设子系统A负责自动驾驶相关应用，子系统B负责智能座舱相关应用。由于智能座舱对NPU的需求较少，而自动驾驶对NPU的需求较高。故在子系统A端Linux域跑的自动驾驶应用可以通过RPC调用方式调用子系统B端Android域闲置的NPU设备。

不同SOC之间的调度

关于两个X9U芯片之间的调度由于外部网卡与内部的虚拟网卡不在同一网段内，所以两个芯片之间只有分配了外部网卡的域之间可以直接RPC连接，其它域由于经过了虚拟网卡，所以需要通过代理模式连接。
